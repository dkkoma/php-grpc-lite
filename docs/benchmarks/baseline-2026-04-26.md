# ベンチマーク・ベースライン: Phase 0 完了時点

> **取得日**: 2026-04-26
> **コミット**: Phase 0 完了直後
> **目的**: Phase 1 以降(C 拡張化、接続プーリング等)の改善・退行を測る原点を固定する

---

## 1. 環境

| 項目 | 値 |
|---|---|
| ベースイメージ | `php:8.4-cli-trixie` |
| PHP | 8.4.20 (NTS) |
| ext-curl, ext-protobuf, ext-openssl | 有効 |
| アーキテクチャ | linux/arm64(Apple Silicon ホスト上の Docker) |
| ネットワーク | docker compose 内部ネットワーク(loopback 同等) |
| 対向サーバ | `test-server`(Go gRPC, h2c, port :50051) |

ベンチは全て自前 Go 製 test-server に対して実行する。`BenchUnary` / `BenchServerStream` RPC を備えており、payload bytes、message count、per-message server delay を request 1 つで指定できる。これによりラン間 ±2-3% 程度の安定指標が得られる。

## 2. 計測手法

PHPBench 1.6 を使用。各シナリオで:

- 各 iteration で revs 回(= 同じ呼び出しを連続)発行 → 平均をその iteration の値に
- iterations を 5 回繰り返して mode(最頻値)を取る
- 1 iteration を warmup として捨てる

リクエストは `setUp` で構築済みのインスタンスを使い回し、計測対象は **gRPC 呼び出しと `wait()` / `responses()` の iteration** のみ(オブジェクト生成コストを含まない)。

## 3. 結果

### 3.1 Unary

| ベンチ | サブジェクト | パラメータ | mode | RSD |
|---|---|---|---:|---:|
| `UnaryLatencyBench` | `benchSayHello` | — | **228 μs** | ±1.6% |
| `UnaryPayloadBench` | `benchBenchUnary` | payload=0 | 229 μs | ±1.9% |
| | | payload=100 B | 221 μs | ±2.6% |
| | | payload=1 KB | 243 μs | ±3.4% |
| | | payload=10 KB | 250 μs | ±1.3% |
| | | payload=100 KB | **384 μs** | ±1.2% |

### 3.2 Unary with server delay

| ベンチ | サブジェクト | mode | RSD |
|---|---|---:|---:|
| `UnaryDelayBench` | `benchSingleCallWithServerDelay10ms` | **17.2 ms** | ±1.3% |
| `UnaryDelayBench` | `benchSequentialTenCallsServerDelay10ms` | **174 ms** | ±2.0% |

(server が `time.Sleep(10ms)` してから応答する)

### 3.3 Server streaming

#### 3.3.1 メッセージ数を振る(payload=100 B、delay=0)

| 件数 | mode | RSD |
|---:|---:|---:|
| 10 | 240 μs | ±1.9% |
| 100 | 344 μs | ±2.1% |
| 1000 | **1.38 ms** | ±2.6% |

#### 3.3.2 ペイロードサイズを振る(count=100、delay=0)

| サイズ/msg | 総バイト | mode | RSD |
|---:|---:|---:|---:|
| 0 B | 0 | 317 μs | ±1.9% |
| 100 B | ~10 KB | 614 μs | ±2.5% |
| 1 KB | ~100 KB | 752 μs | ±2.2% |
| 10 KB | ~1 MB | **1.55 ms** | ±1.1% |

#### 3.3.3 サーバ pacing 経由(count=10、payload=100 B)

| メッセージ間 delay | mode | 期待値(server 待ち分) |
|---:|---:|---:|
| 0 ms | 512 μs | — |
| 1 ms | 20.3 ms | ~9 ms (= 9 × 1ms) |
| 10 ms | 107 ms | ~90 ms (= 9 × 10ms) |

incremental yield が server cadence にぴったり追従していることが定量的に確認できる(delay × 9 + 接続/decode オーバヘッド ≒ 計測値)。

### 3.4 派生指標

- **Unary per-byte**: `(384 − 229) μs / 100 KB ≈ 1.5 ns/byte`(framing + protobuf decode + libcurl の合計)
- **Streaming per-message**: `(1.38 ms − 240 μs) / 990 ≈ 1.15 μs/msg`(curl_multi loop + frame 再構成 + decode + Generator yield)
- **Streaming per-byte**: `(1.55 ms − 317 μs) / (100 × 10 KB) ≈ 1.2 ns/byte`

### 3.5 メモリ

ピーク常駐メモリは全シナリオで 1.9 - 2.0 MB。

## 5. 再現方法

```bash
# 我々の実装単独
docker compose run --rm dev vendor/bin/phpbench run --report=aggregate

# ext-grpc との両側比較
./bench/compare.sh
```

タグ付けして履歴を残す:

```bash
docker compose run --rm dev vendor/bin/phpbench run --report=aggregate --tag=baseline-2026-04-26
docker compose run --rm dev vendor/bin/phpbench run --report=aggregate --ref=baseline-2026-04-26
```

(`.phpbench/storage` に履歴が積まれる。本リポジトリでは gitignore 済み)
